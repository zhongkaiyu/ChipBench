# src/run_verification.py & main.py

Build pipeline and CLI for cross-language verification.

## CLI usage (main.py)

```bash
# All three DUTs
python main.py ref.sv --dut-sv dut.sv --dut-cc dut.cc --dut-py dut.py

# Only Verilog DUT
python main.py ref.sv --dut-sv dut.sv

# Only CXXRTL + Python
python main.py ref.sv --dut-cc dut.cc --dut-py dut.py

# With JSON port description
python main.py ref.sv --dut-sv dut.sv --json ports.json

# Custom work directory
python main.py ref.sv --dut-sv dut.sv -w /tmp/build
```

| Argument | Description |
|---|---|
| `ref_sv` | Golden reference Verilog (module `RefModule`). Required. |
| `--dut-sv` | DUT Verilog (module `TopModule`) |
| `--dut-cc` | CXXRTL C++ file |
| `--dut-py` | Python DUT file (class `TopModule`) |
| `--json` | JSON port description (alternative to parsing ref.sv) |
| `-w` | Working directory (default: `work`) |

At least one DUT must be provided.

## Pipeline (run_verification.py)

### `run_verification(ref_sv, dut_sv=None, dut_cc=None, dut_py=None, json_file=None, work_dir="work") -> int`

Returns `0` on success, `1` on failure.

### Pipeline steps

```
1. Copy files to work directory
2. Generate testbench.cpp (via generate_testbench())
3. Compile with Verilator
4. Build and link
5. Run simulation
```

### Build flow

The build adapts based on which DUTs are provided:

**With `dut_sv`:**
1. `verilator --cc ref.sv` → VRefModule
2. `verilator --cc dut.sv --exe testbench.cpp` → VTopModule
3. Build both libraries, link together

**Without `dut_sv`:**
1. `verilator --cc ref.sv --exe testbench.cpp` → VRefModule + testbench
2. Build and link with VRefModule only

CXXRTL is `#include`d in `testbench.cpp` — not compiled separately. The CXXRTL include path is added to CFLAGS only when `dut_cc` is provided.

Python runs via subprocess at simulation time — no build step needed.

### Module naming conventions

| File | Required name |
|---|---|
| `ref.sv` | `module RefModule` |
| `dut.sv` | `module TopModule` (can contain multiple modules) |
| `dut.cc` | `cxxrtl_design::p_TopModule` (generated by `yosys -p "write_cxxrtl"`) |
| `dut.py` | `class TopModule` with `eval()` method (can contain multiple classes) |

### Python DUT protocol

Each `eval()` call = one clock cycle (posedge). The `clk` signal is **not** passed in inputs — it's managed by the C++ testbench. Only non-clock signals are passed.

```python
class TopModule:
    def __init__(self):
        self.state = 0

    def eval(self, inputs):
        # inputs: {"rst_n": 1, "data_in": 42, ...}
        # Each call = one posedge
        if inputs.get("rst_n", 1) == 0:
            self.state = 0
        else:
            self.state = ...
        return {"Q": self.state}
```
