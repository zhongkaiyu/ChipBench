  - `stallreq_from_if`: Stall request signal from the instruction fetch stage.
  - `stallreq_from_id`: Stall request signal from the instruction decode stage.
  - `stallreq_from_ex`: Stall request signal from the execution stage.
  - `stallreq_from_mem`: Stall request signal from the memory stage.

- **Output ports:**
  - `new_pc [31:0]`: Output for the new program counter value, updated during exceptions.
  - `flush`: Signal to flush the pipeline.
  - `stall [5:0]`: Stall control signal for the pipeline.

   - `flush` should be set to `0`.

   - System call (`excepttype_i == 32'h00000008`): Set `new_pc` to `32'h00000040`.
   - Invalid instruction (`excepttype_i == 32'h0000000a`): Set `new_pc` to `32'h00000040`.
   - Trap (`excepttype_i == 32'h0000000d`): Set `new_pc` to `32'h00000040`.
   - Arithmetic overflow (`excepttype_i == 32'h0000000c`): Set `new_pc` to `32'h00000040`.
   - ERET (`excepttype_i == 32'h0000000e`): Set `new_pc` to the value of `cp0_epc_i`.

3. If no exception occurs, the module should handle stall requests from different stages:
   - If `stallreq_from_mem` is asserted, set `stall` to `6'b011111` and do not flush the pipeline.
   - If `stallreq_from_ex` is asserted, set `stall` to `6'b001111` and do not flush the pipeline.
   - If `stallreq_from_id` is asserted, set `stall` to `6'b000111` and do not flush the pipeline.
   - If `stallreq_from_if` is asserted, set `stall` to `6'b000111` and do not flush the pipeline.

4. If no stall request or exception occurs, the outputs should remain in their default state:
   - `stall = 6'b000000`
   - `flush = 0`
   - `new_pc = 0`

Make sure to implement this functionality in a clean and modular way, using combinational or sequential logic as appropriate.
